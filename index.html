Executive summary

What the code does and what it produced

ExoMetrix trains a two-stage ML pipeline for real-time gait analysis. The workflow reads raw goniometer (joint angle) data from the Camargo et al. AB subject biomechanics dataset, extracts knee-angle signals from MATLAB files, down-samples to a fixed maximum length, calibrates to a clinical range, and trains three models that ship together as a single .joblib artifact. The same run also produces evaluation plots and a metadata JSON for the API status endpoint.

What this documentation covers in detail: discovery rules, signal extraction fallbacks, calibration math, feature definitions (including stride-normalized velocity), label heuristics, model training (holdout + cross-validation), evaluation outputs, and export format.

Step 1 — argument parsing and path setup

The CLI accepts --dataset-root, --subject-glob, --output-model, and --output-meta. Defaults assume the training script lives two levels below the dataset root. Paths are resolved into absolute locations for the model artifact and metadata JSON.

Step 2 — subject discovery

Checks whether dataset_root is already a single subject folder (detected by looking for an osimxml/ child) or a parent of multiple subjects. If it is a parent, the code globs for AB* folders. As a final fallback, it searches for any folder that contains gon/*.mat files. This supports AB06-only training or multi-subject training without manual path edits.

Step 3 — MAT loading

Each gon/*.mat file is loaded with scipy first, then h5py as a fallback for MATLAB v7.3 HDF5 files. This ensures compatibility across older and newer MATLAB formats.

Step 4 — signal extraction and downsampling

Signal extraction walks every numeric array in the MATLAB structure, scores candidates by variance and dynamic range, and selects the most angle-like series. If no candidate is found, the function attempts to decode the MATLAB function workspace blob as raw float64 data. Signals are filtered to the plausible angle range (-220 to 220) and downsampled to at most 2000 samples using stride = max(1, len(signal) // 2000). The downsampling stride is saved and used later to normalize velocity features.

Step 5 — global calibration

All subjects' angles are pooled and linearly rescaled so the 1st–99th percentile range maps to 5 to 120 degrees (typical clinical gait range). Values are hard-clipped to 0 to 180 degrees. This keeps the anomaly model on a clinically meaningful scale and makes subject data comparable.

Step 6 — anomaly model training

Calibrated angles are reshaped to (N, 1) and standardized with RobustScaler. An IsolationForest with 300 trees and 8% contamination is trained. The decision score distribution is summarized and percentiles (p01, p05, p50, p95, p99) are stored for runtime normalization.

Step 7 — window features and labels

The script iterates the MAT files a second time. Each file's full signal is treated as one window and 14 statistical features are computed. Velocity features use np.diff() divided by the saved downsampling stride, which stabilizes scale across files. Labels come from the file path and filename heuristics.

Step 8 — classifier training and baseline

Both RandomForest classifiers receive the same feature matrix. An 80/20 stratified split validates accuracy when all classes have at least 2 samples. A most-frequent baseline model is trained on the same split to show the minimum performance a trivial classifier would achieve.

Step 9 — cross-validation and plots

For activity classification, StratifiedKFold cross-validation is computed (up to 5 folds based on the smallest class). A learning curve is generated from the same folds. Additional plots include a confusion matrix, feature correlation heatmap, PCA projection, feature importance, and label distributions.

Step 10 — artifact saving

Everything is packed into a single dict and saved with joblib.dump(). A JSON metadata file is also written for the Flask backend status endpoint. Metadata includes counts, label distributions, validation and baseline accuracy, cross-validation mean and standard deviation, and a flag showing velocity normalization.

    Feature engineering

    The 14 statistical features computed per window

    Each MAT file is treated as one signal window. 14 statistics are extracted: 8 positional, 4 velocity-based, 1 energy, and 1 temporal structure metric. Velocity is computed as first differences and then divided by the downsampling stride to keep per-file scales consistent.

    mean_angle
    μ = (1/N) Σ xᵢ
    Average joint position — separates treadmill (low) from stair (high)
    std_angle
    σ = √Var(x)
    Motion spread. High for stairs, low for constrained treadmill
    min_angle
    min(x)
    Minimum joint position during window
    max_angle
    max(x)
    Stair needs high max; asymmetry encodes ramp
    range_angle
    max(x) − min(x)
    Total arc. Reduced range may indicate stiffness
    median_angle
    x̃ = pct(x, 50)
    Robust central tendency — differs from mean on skewed ramp signals
    p10_angle
    pct(x, 10)
    Robust lower bound, resistant to outliers
    p90_angle
    pct(x, 90)
    Robust upper bound, resistant to outliers
    mean_abs_velocity
    mean(|dx|) / stride
    Average angular speed. Running → high; slow walking → low
    std_velocity
    std(dx / stride)
    Speed irregularity. High for stairs, low for treadmill
    max_velocity
    max(dx / stride)
    Peak forward angular acceleration
    min_velocity
    min(dx / stride)
    Peak reverse angular acceleration
    energy
    (1/N) Σ (xᵢ−μ)²
    Actually variance — redundant with std² but kept for naming clarity
    peak_density
    peaks / N, t=μ+0.25σ
    Approximate step frequency. Irregular gait → unusual density

    Note Velocity features are normalized by the downsampling stride. This makes per-file velocity values comparable, but still not true time-based velocity because the real sampling rate is not passed into training.

    Labeling and dataset

    Where labels come from and where they may be wrong

    Labels come entirely from file system paths — there is no human-annotated label file. The grandparent folder name gives the activity, and the filename stem provides intent signals.

    stair/ stair upstairs Always upstairs — descent not captured

    Bias and noise concerns

    HighStair and ramp descent are absent
    All stair files → "upstairs", all ramp files → "ramp_up" regardless of actual direction. A patient descending stairs will be confidently misclassified.
    MediumRunning detection is fragile
    Only files with _fast_ in the filename get labelled as running. Renamed files silently become "walking". Only 31 windows currently — the class is thin.
    LowHealthy young adult population only
    3 able-bodied subjects from the AB cohort. No pathological gait, elderly movement, or post-surgical compensation in training data.

    Model training decisions

    Why each algorithm and parameter was chosen

    Why RobustScaler?

    Gait angle data contains real physiological outliers — sudden jerks, sensor spikes, non-physiological MATLAB fill values. RobustScaler uses the median and IQR instead of mean and std, so spike values don't distort the scale. StandardScaler would pull the scale toward outliers and make the IsolationForest treat normal gait as unusual.

    Why IsolationForest? Why 8% contamination?

    IsolationForest works by randomly splitting features and measuring how quickly a point can be isolated. Genuine outliers get isolated in very few splits; normal inliers need many. It suits this case because: training data has no anomaly labels, it's fast for embedded deployment, and it handles the asymmetric joint-angle distribution well.

    The contamination=0.08 value tells the model to expect ~8% of training samples to be outliers. Since the AB subjects are healthy, this effectively flags "8% of healthy gait as edge-cases worth flagging." A lower value (0.03–0.05) would reduce false positives on real patients.

    Why RandomForest? Why 350 trees?

    RandomForest is robust for small datasets (434 windows) and provides probability estimates needed for confidence scoring. class_weight="balanced_subsample" resamples class weights at each tree independently, handling the ramp/treadmill imbalance. 350 trees is higher than needed — diminishing returns likely set in around 100–150 with 434 samples and 14 features.

    When is validation skipped?

    Validation is skipped when total samples < 30 or any single class has fewer than 2 members (stratified split requires ≥2 per class in both sets). In the current snapshot neither applies, so validation was performed — 87-window holdout, single split, 93.1% accuracy.

    Cross-validation and baseline

    StratifiedKFold cross-validation is run on activity labels to estimate stability across folds. This is still a window-level split (not leave-one-subject-out), so results may be optimistic for new subjects. A most-frequent baseline classifier is also reported to show how much the RandomForest improves over trivial prediction.

    The anomaly score normalization

    Raw IsolationForest scores have no natural 0–1 scale. The code stores training score percentiles p01 and p05. At runtime: anomaly_strength = clip((p05 − score) / (p05 − p01), 0, 1). Current values: p01=−0.113, p05=−0.027 → denominator ≈ 0.086. This narrow band makes scores near the boundary sensitive to small fluctuations.

    Relationship between anomaly model and classifiers

    The two stages are architecturally independent but semantically complementary. The IsolationForest scores a single angle per call: "is this angle consistent with normal gait?" The RandomForests score a 14-feature window: "what kind of activity is this?" A compensating step can occur during any activity — the classifiers still assign the correct activity even when the anomaly detector fires. Both run in parallel inside predict().

    Audience summaries

    The same system explained two different ways

    For clinicians

    ExoMetrix learns what normal walking, stair climbing, and ramp walking look like from joint angle data collected on healthy young adults. When a patient uses the system, it compares their joint angles against this learned normal and flags unusual patterns as "compensating steps." It also estimates the type of movement the patient is doing and how much assistance the exoskeleton should provide.

    Currently the system has been trained on only three healthy volunteers. It has not been tested on patients recovering from injury or surgery. Estimates for toe clearance and walking phase are approximations, not clinical measurements. Before clinical deployment, the system should be validated against real patient data and outputs - particularly step quality - should be reviewed by a clinician familiar with the patient population.

    For engineers

    Two-stage architecture: unsupervised anomaly detector (IsolationForest on univariate calibrated angle, RobustScaler) trained on 868K samples, paired with two supervised classifiers (RandomForest, 350 trees, balanced_subsample) trained on 14 hand-crafted stats per-file window (434 total). Labels are path-derived — no annotation file.

    Main technical risks: (1) one vector per file vs. the 120-sample runtime window — training and inference features are computed at different temporal scales; (2) velocity is normalized by downsampling stride but still lacks true time normalization (no Hz); (3) anomaly score normalization band is narrow and noisy near the boundary; (4) cross-validation is window-level, not leave-one-subject-out. Priority fixes: sliding-window extraction, subject-wise validation, explicit sample-rate handling, and descent locomotion labels.
